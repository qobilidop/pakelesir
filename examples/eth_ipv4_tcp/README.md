# Example: `eth_ipv4_tcp`

Everything in this directory is generated from **one description**
([the builder code](../../src/examples.rs)) and equality-guarded by
tests — if any file here drifts from what the toolchain generates, CI
fails. Regenerate with `./dev.sh cargo run --bin gen_examples`.

| File | What it is | Generated by |
|---|---|---|
| [`ir.json`](ir.json) | The description itself, as normative Pakeles IR (protojson) | `pakeles export-ir` |
| [`dissector.lua`](dissector.lua) | Working Wireshark dissector (Lua 5.2) | `pakeles gen lua` |
| [`doc.md`](doc.md) | Field tables + parse graph documentation | `pakeles doc` |
| [`graph.dot`](graph.dot) / [`graph.svg`](graph.svg) | The parse graph | `pakeles viz` |
| [`vectors.json`](vectors.json) | Path-complete conformance suite: 164 solver-derived test vectors (11 accept / 17 reject / 136 truncation) | `pakeles testgen` |
| [`vectors.pcap`](vectors.pcap) | The 28 byte-aligned vectors as a capture file | `pakeles testgen --pcap-out` |

Try the dissector against the vectors, no build required — any machine
with Wireshark ≥ 4.x:

```sh
tshark -X lua_script:dissector.lua -r vectors.pcap -V
```

It registers as a postdissector, so its tree appears alongside
Wireshark's built-in dissection. The generated dissector's output has
been verified inside tshark to agree with the reference interpreter on
every vector in `vectors.json`.
